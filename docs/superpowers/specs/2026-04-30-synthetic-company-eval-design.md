# Synthetic Company Corpus for Eval Sets — Design

**Status:** Draft
**Date:** 2026-04-30
**Author:** Mahit Namburu
**Branch:** `feat/synthetic-eval-corpus`

## 1. Summary

A tool that takes one or more code repositories (and an optional thin
"company context" YAML), and produces a **complete, internally consistent
synthetic company corpus** spanning every connector prbe-knowledge supports
— Slack, Linear, GitHub, Notion, Sentry, Granola, Claude Code. The corpus
flows through the real ingestion pipeline (R2 → `ingestion_queue` → real
worker → real normalizer/chunker/embedder/graph), so the resulting
documents/chunks/edges live in the same DB shape as production data and can
be queried by the same retrieval service.

Two lightweight ground-truth artifacts come out of each run as a byproduct
(`questions.jsonl`, `manifest.json`); richer eval-scoring infrastructure is
intentionally out of scope and lands in a follow-up.

This supersedes `scripts/seed_synthetic.py`.

## 2. Goals

1. **Plug in repos, get a corpus.** One CLI invocation with a profile YAML
   produces a multi-source eval-grade corpus against a fresh tenant.
2. **Multi-repo first-class.** A "company" is the union of multiple repos
   and the people / services / dependencies they imply.
3. **Realistic, not random.** Every doc is part of a real-shaped narrative
   (incident, refactor, weekly meeting, hiring pipeline, etc.). No "noise"
   tier; corpus texture comes from archetype diversity, not from random
   generation.
4. **Cross-source coherence.** The Slack thread, the Linear ticket, the
   Notion postmortem, and the GitHub PR about the same incident are
   mutually consistent — same cast, same dates, same root cause.
5. **Reproducible.** Same profile + same repo SHAs + warm caches → byte-
   identical R2 keys and queue rows.
6. **Direct-to-storage.** No real provider connections, no webhook
   replay. Synthetic raw envelopes drop directly into R2 + `ingestion_queue`
   and the existing worker handles the rest.

## 3. Non-goals

- **Real-provider replay** (creating actual Slack messages, Notion pages,
  Granola notes). Different test category, different infra, future work.
- **Webhook signature exercise** (POST to `prbe-backend/webhooks/{source}`).
  Future mode; not v1.
- **Detailed eval scoring runner.** v1 emits `questions.jsonl` in the
  format the existing `tests/evals/run.py` stub already expects. A richer
  structured `eval_set.json` is future work.
- **Live / real-time generation.** Synth runs are batch; the corpus is
  fully written before evals run against it.
- **Cross-tenant data sharing.** Each profile creates one synthetic tenant
  in isolation.

## 4. Vocabulary

| Term | Definition |
|---|---|
| **WorldModel** | Deterministic, immutable structure derived from input repos: people, services, dep graph, topic pool, channels, time anchors. The "setting." |
| **CompanyContext** | Optional thin YAML (or auto-inferred) capturing non-code business reality: customers, non-eng personas, market, recent milestones. Augments WorldModel. |
| **Archetype** | A hand-coded plot shape (e.g. `INCIDENT`, `WEEKLY_ENG_REVIEW`, `ROADMAP_INITIATIVE`). Library of ~12-20. |
| **ScenarioSpec** | A structured per-scenario instance of an archetype: cast, timeline, root cause, source emissions, eval questions. Generated by Planner LLM. |
| **SourceWrapper** | Per-source deterministic envelope shaper. Takes generated content, produces byte-exact provider webhook envelope. |
| **Profile** | User YAML driving a single run: customer_id, repos, preset/archetype mix, seed. |
| **Preset** | Canonical archetype-mix recipe shipped with the tool (e.g. `flagship`, `incidents-only`). |
| **Run** | One invocation of `synth run`. Produces a directory of artifacts under `eval-datasets/<run-id>/`. |

## 6. Deterministic layer

### 6.1 RepoExtractor

Per-repo, returns `RepoSignals`. Hybrid: local clone is mandatory, GitHub
API token is optional.

```python
@dataclass
class RepoSignals:
    url: str
    clone_path: Path
    default_branch: str
    latest_sha: str                   # cache key
    description: str | None
    topics: list[str]

    # Local-only
    file_tree: FileTree
    manifests: list[Manifest]         # pyproject/package.json/Cargo/go.mod/fly.toml
    readmes: list[Readme]             # root + per-dir; section-split
    codeowners: list[CodeownerRule]
    commits: list[Commit]             # last 90d, files_touched, author identity
    branches: list[Branch]
    compose_services: list[str]

    # GitHub-API (None when no token)
    issues: list[Issue] | None
    prs: list[PR] | None
    contributors: list[Contributor] | None
    workflows: list[Workflow] | None
```

Cached to disk by `(repo_url, latest_sha)` — re-running on the same
checkout is free.

GitHub API budget: target <500 req/repo (commits via `since`, issues/PRs
capped at 200 each, file contents read from local clone not API). Hard
limit 1000 req/repo.

### 6.2 WorldModelMerger

Merges `RepoSignals[]` into one immutable `WorldModel`.

```python
@dataclass(frozen=True)
class WorldModel:
    repos: list[RepoSummary]
    people: list[Person]              # canonicalized across repos
    services: list[Service]
    topic_pool: list[Topic]
    channels: list[ChannelHint]       # synthesized Slack channels
    notion_sections: list[SectionHint]
    time_anchors: TimeAnchors
    dep_graph: list[DepEdge]
    company_name: str
    seed: int
    extracted_at: datetime
    sha_set: dict[str, str]           # repo_url → sha
```

**Person canonicalization** is the highest-stakes merge step (failure mode:
treating two people as one produces threads where someone replies to
themselves):

1. Build identity index from each repo:
   - GitHub contributors: `gh_username → {display_name, email[]}`
   - Git commits: `(author_email, author_name)` pairs
2. Canonical-key precedence:
   - `gh_username` if present anywhere for this email/name
   - Exact email match
   - **Never** merge by display_name alone (too many "John"s)
3. Tally per-repo activity for each canonical id; drop personas with
   `activity_score < min_threshold` (default: 2 commits).
4. Cap at top N personas (default 25; configurable in profile).

**Service inference + collision**:

- Per-repo candidates: the repo itself (single-purpose), manifest names,
  children of `services/`/`apps/`/`packages/`/`cmd/`, fly.toml app names,
  docker-compose service names.
- Type inference: Dockerfile + HTTP → `api`; `asyncpg.listen` → `worker`;
  `next.config.js`/Vite → `frontend`; `[bin]` in Cargo.toml → `cli`.
- Cross-repo same-name: if a manifest dep edge exists, dedupe to the
  implementation's repo. Else namespace as `repo/svc`.
- Owner resolution: walk CODEOWNERS, resolve globs, map handles to
  canonical Person ids; union across repos.

**Topic pool weighting**:

```
weight(topic) = recency_decay(topic.ts)        # half-life 30d
              × source_weight[topic.kind]       # pr=1.0, issue=0.8, commit=0.5,
                                                # readme_section=0.3, branch=0.4
              × repo_activity_norm(topic.repo)
              × (1 + log(1 + len(mentioned_services)))
```

Plot-heavy archetypes draw topics with replacement (weighted); recurring/
cadence-driven archetypes draw without replacement to spread coverage
across distinct topics.

**Cross-repo dependency edges**: parse manifest deps; if `dep.name` matches
a `Service.name`, record `DepEdge(from→to)`. Lets scenarios span repos
coherently ("breaking change in lib X (repo B) caused incident in service
Y (repo A)").

**Channel + Notion section synthesis** (rules, no LLM):

```
slack_channels =
    {f"#{svc.name}" for svc in services if svc.kind in (api, worker, frontend)}
  ∪ {f"#{svc.name}-deploys" for svc in top_5_services_by_activity}
  ∪ {f"#team-{team}" for team in distinct_codeowner_teams}
  ∪ generic_channels  # #general, #random, #incidents, #engineering

notion_sections =
    fixed_set: {Engineering, Runbooks, Postmortems, Architecture,
                Onboarding, Product, People & Hiring}
  ∪ {f"{svc.name} runbook" for svc in top_10_services}
```

IDs are deterministic hashes; re-runs produce stable IDs.

**Privacy default**: emails are kept as-is. Real coworker linkage is part
of corpus realism. Synth tenants are isolated by `customer_id` prefix and
cleanable in one command, so PII risk is bounded to the run.

### 6.3 CompanyContext

Optional input. Closes the gap between repo signals and business reality
(customers, non-engineering personas, strategic priorities, recurring
cadences). When absent, a single LLM call over aggregated READMEs +
repo descriptions produces a draft, written to
`eval-datasets/<run-id>/inferred-company.yaml` for the user to inspect.

```yaml
# company.yaml
name: prbe.ai
stage: seed
headcount: 8
market: AI-native knowledge for engineering teams
competitors: [Glean, Notion AI, Slack AI]
customers:
  - {name: Acme Corp, type: design_partner, stage: pilot}
  - {name: Globex, type: paying, plan: team}
non_eng_people:
  - {name: Sam Park, role: founding GTM, slack: U_SAM}
recent_milestones:
  - Shipped MCP integration Q1
  - Closed seed extension March
ongoing_initiatives:
  - SOC2 Type 1
  - Onboarding flow v2
cadence:
  weekly: [eng-review-mon, all-hands-fri]
  biweekly: [product-review]
```

CompanyContext is consumed by **both plot-heavy and recurring archetypes**;
the prompt-cache hit rate makes the cost difference negligible.

## 7. Archetype library

Single library; archetypes vary by **narrative density** and **stakes**.
No separate "chatter" tier — every doc lives inside an archetype instance.

```python
@dataclass
class Archetype:
    name: str
    category: Category                  # plot | recurring | slow_burn
    cadence: Cadence | None             # for recurring: WEEKLY|BIWEEKLY|MONTHLY|SPRINT
    docs_per_instance: tuple[int, int]
    sources_used: list[Source]
    cast_size: tuple[int, int]

    # Rigor knobs
    needs_planner_call: bool            # plot: yes; many recurring: no (templated)
    validator_level: ValidatorLevel     # strict | name-only | none
    eval_questions: EvalDifficulty | None  # None = doesn't contribute ground truth

    spec_template: str                  # Planner prompt template
    writer_voice_traits: list[str]      # archetype-specific tone hints
```

### 7.1 V1 archetype roster

**Plot (closed narrative, eval-bearing):**

| Archetype | Sources | Sample question |
|---|---|---|
| `INCIDENT` | sentry, slack, linear, notion, github | "What caused the {svc} outage on {date}?" |
| `CUSTOMER_ESCALATION` ∗ | slack, linear, granola | "What's the status of {customer}'s issue?" |
| `LAUNCH` | slack, notion, granola, linear | "When did {feature} ship and who owned it?" |
| `BIG_REFACTOR` | github, slack, notion, linear | "Why did we move {X} from {A} to {B}?" |
| `PERF_REGRESSION` | sentry, slack, github | "What regressed {svc} p95?" |
| `DEPENDENCY_BUMP` | github, slack, linear | "Why did we upgrade {lib}?" |

**Recurring (cadence-driven, light validator, optional eval questions):**

| Archetype | Cadence | Sources |
|---|---|---|
| `WEEKLY_ENG_REVIEW` | weekly | granola, slack, notion |
| `SPRINT_PLANNING` | biweekly | granola, linear, notion |
| `RETRO` | biweekly | granola, notion |
| `ALL_HANDS` | weekly | granola |
| `ON_CALL_HANDOFF` | weekly | slack, notion |
| `DEPLOY_THREAD` | per-deploy | slack |
| `PR_REVIEW` | per-PR | github |
| `STANDUP_UPDATE` | daily | slack |

**Slow-burn (multi-week arcs, medium-difficulty questions):**

| Archetype | Span | Sources |
|---|---|---|
| `ROADMAP_INITIATIVE` | 30d | notion, granola, slack |
| `ARCH_RFC` | 7d | notion, slack, granola |

| `DEBATE` | 3-7d | slack, notion, github |

∗ = requires CompanyContext (customers, non_eng_people, etc.); planner skips
otherwise.

### 7.2 Why this is enough information

Scenarios feel real because each is grounded in:
- Real services with real owners (CODEOWNERS-derived)
- Real technical specifics (file paths, PR numbers, deploy targets)
- Real recent topics (commits, PRs, issues)
- Real dependencies (cross-repo edges enable cross-repo plots)
- Realistic timing (commit-cluster anchors)

What's *invented* is the connective tissue: the incident's exact root cause
story, the debate's positions, the postmortem's recommendations. This is
exactly what eval questions test retrieval/synthesis of, and the synth tool
controls what's in the corpus, so the ground truth is provable.

## 8. Narrative layer

### 8.1 ScenarioRunner

Walks the archetype library according to profile config. For each archetype
× count (or × cadence ticks across `time_window`), instantiates a scenario:

```
For archetype A in profile.archetypes:
    instances = compute_instances(A, profile.time_window)
    for instance in instances:
        if A.needs_planner_call:
            spec = Planner.plan(A, world, company_context, instance.ts)
        else:
            spec = TemplatedSpec.build(A, world, instance.ts)  # deterministic
        validate_spec(spec, world)                              # reject + retry up to 2x

        for source in spec.source_emissions:
            doc = Writer.write(spec, source, prior_emitted_docs)
            yield doc

        run_validator(scenario_docs, spec, A.validator_level)
        emit_eval_questions(spec, A.eval_questions)
```

### 8.2 Planner

One LLM call per scenario (skipped for low-rigor archetypes). Outputs
structured `ScenarioSpec`, no prose:

```python
ScenarioSpec {
    id, archetype, title, summary
    cast: [{ person_id, role_in_scenario }]
    affected_services, affected_repos
    root_cause / outcome / decision (archetype-shaped)
    timeline: [{ ts, source, kind, channel? }]
    source_emissions: { source: count }
    eval_questions: [{ q, expected, evidence_doc_keys, difficulty }]
}
```

**Spec validation** (deterministic, no LLM): all `person_id`/`service_id`
exist in WorldModel; timeline monotonic; every `evidence` key matches a
planned emission. Reject + retry up to 2×; drop scenario after.

### 8.3 Writer

Per scenario × per source: ~6 LLM calls per plot scenario, fewer for
recurring. Generates source-shaped content (Slack messages, Notion blocks,
Granola transcript, GitHub PR body).

Critical detail: **the writer sees earlier-emitted docs from the same
scenario, but only the current persona's view of them.** Bob writing the
postmortem sees the Slack thread; Alice writing her first responder Slack
message sees nothing prior. Prevents future-knowledge leak into earlier
docs.

### 8.4 Validator

Two passes:

**Pass 1 (deterministic)**:
- Extract entity mentions from generated docs (regex over WorldModel
  service/people/channel names)
- Assert dates within scenario window
- Assert services mentioned ⊆ `scenario.affected_services ∪ adjacent_deps`
- Assert people mentioned ⊆ `scenario.cast ∪ small_random_extras`
- Hallucination check: any service/person name *not* in WorldModel? Flag.

**Pass 2 (cheap LLM, one call per scenario)**:
- "Given this scenario spec, do these N docs tell a consistent story?"
- Output: `{pass: bool, violations: [{doc_id, issue}]}`

If violations on > 30% of docs OR any hallucination: regenerate failing
docs with a corrective prompt. After 2 regen rounds, drop the scenario from
output and log to `manifest.warnings`.

**Validator level per archetype:**
- `strict`: both passes (plot archetypes)
- `name-only`: pass 1 only (recurring archetypes)
- `none`: skipped (high-confidence templated archetypes; rare)

### 8.5 Confabulation control

1. **Allowlist in writer prompts** — every prompt explicitly lists allowed
   service/people/channel names; instructs "do not invent names not on this
   list, except for incidental third-party SaaS."
2. **Post-extract verification** — entity extractor flags out-of-allowlist
   proper nouns; cheap LLM classifier decides if it's third-party SaaS
   (allowed) or invented service (rejected).
3. **Regen on violation** with the violation specifically called out in the
   corrective prompt.

## 9. Output layer

### 9.1 SOURCE_WRAPPERS

Per-source, deterministic, no LLM. Takes generator output and produces the
byte-exact provider envelope the connector expects. One per `SourceSystem`:

| Source | Envelope | Generator's content |
|---|---|---|
| `slack` | `event_callback` with `event:{type:message, channel, user, text, ts, thread_ts?}` | message text + threading hint |
| `notion` | `page.updated` webhook with page metadata + blocks | block array (Notion block JSON) |
| `granola` | `/v1/notes` row | transcript chunks + title + attendees |
| `github` | per-event-type webhook (`pull_request`, `issues`, `issue_comment`, `push`) | per-type structured fields |
| `linear` | GraphQL webhook | issue/comment fields |
| `sentry` | `issue_created` / `event_alert` | issue fields + minimal stack |
| `claude_code` | internal envelope (see `shared/claude_code_extraction.py`) | session/turn records |

Each wrapper is fixture-tested against an existing `fixtures/<source>/`
payload — round-trip through `Connector.parse` must succeed.

### 9.2 IngestionWriter

```python
class IngestionWriter:
    async def write(self, doc: SynthDoc) -> None:
        key = f"raw/{doc.source}/{customer_id}/synth/{doc.source_event_id}.json"
        await store.put(bucket, key, doc.envelope_bytes)
        self.batch.append((doc, key))
        if len(self.batch) >= batch_size:
            await self._flush_queue()

    async def _flush_queue(self) -> None:
        await conn.executemany("""
            INSERT INTO ingestion_queue
              (customer_id, source, source_event_id, raw_key, status,
               priority, occurred_at, enqueued_at)
            VALUES ($1,$2,$3,$4,'pending',$5,$6,NOW())
            ON CONFLICT (source, source_event_id) DO NOTHING
        """, [...])
```

Three guarantees:
- **Idempotent.** `(source, source_event_id)` unique constraint + `ON
  CONFLICT DO NOTHING`. R2 puts are key-deterministic. Re-runs are safe.
- **Backpressure-friendly.** Batch 50; worker dequeues at its own rate.
- **Ordering.** Batch insertion preserves emission order so scenario
  timeline events keep relative order in the queue.

### 9.3 Tenant lifecycle

Three idempotent CLI commands:

```bash
synth init  --profile <path>    # bootstrap tenant
synth run   --profile <path> [--reset] [--dry-run]
synth clean --customer <id>     # wipe synthetic tenant
```

`init`:
```sql
INSERT INTO customers (customer_id, display_name, api_key_hash)
  VALUES ($1, 'synth-' || $1, 'synth')
  ON CONFLICT DO NOTHING;
```
Then `store.ensure_bucket(...)`. Then for each source the profile uses,
register a stub row in `integration_tokens` (so the normalizer's token-
lookup path doesn't blow up for sources that require it: granola, notion,
linear).

`clean` is the dangerous one — hard-guarded by customer prefix:
```python
if not customer_id.startswith(("cust-eval-", "cust-synth-")):
    raise ValueError(f"refuse to clean non-synthetic customer: {customer_id}")
```
Then transactional DELETE from `ingestion_queue`, `chunks`, `documents`,
`graph_edges`, `graph_nodes`, `integration_tokens`. Then `store.delete_prefix`
the `raw/` prefix. Customers row left in place as the "tenant exists"
marker.

`--reset` on `run` calls `clean` first.

Synth tenants live in the same DB as real tenants, distinguished only by
`customer_id` prefix. This is intentional — eval data should flow through
the real pipeline and be queried by the real retrieval service.

### 9.4 Eval artifacts

Output directory per run: `eval-datasets/<run-id>/`

```
2026-04-30T14-30Z-flagship-prbe-seed42/
    manifest.json           # run-level summary
    profile.yaml            # frozen resolved profile (preset + overrides merged)
    world_model.json        # debug snapshot
    inferred_company.yaml   # if CompanyContext was auto-inferred
    scenarios/              # one ScenarioSpec per file
    questions.jsonl         # eval-runnable artifact (existing shape)
    docs_index.jsonl        # one row per generated doc; joins to scenarios
    warnings.log            # validator drops, regen rounds, hallucination flags
```

`questions.jsonl` matches `tests/evals/run.py`'s expected
`{input, expected, tags}` shape so the existing stub consumes it without
modification:

```json
{
  "input": "What caused the payments-svc outage on 2026-03-12?",
  "expected": {
    "answer_substring": "feature flag default flipped on without backend ready",
    "evidence_doc_keys": ["raw/slack/.../scn-..-slack-0.json", "raw/notion/.../scn-..-notion-0.json"]
  },
  "tags": ["INCIDENT", "cross-source", "easy"],
  "scenario_id": "scn-incident-payments-2026-03-12",
  "difficulty": "easy"
}
```

Three difficulty bands the planner emits per scenario: `easy` (single-
source recall), `medium-cross-source` (synthesis across 2-3 docs from
different sources), `hard-temporal` (order/timeline reasoning).

`docs_index.jsonl` lets you walk from a query result back to scenario
context:
```json
{
  "doc_id": "scn-incident-payments-2026-03-12-slack-0",
  "scenario_id": "scn-incident-payments-2026-03-12",
  "archetype": "INCIDENT",
  "source": "slack",
  "occurred_at": "2026-03-12T14:05Z",
  "raw_key": "raw/slack/.../scn-...slack-0.json",
  "personas": ["gh:alice", "gh:bob"],
  "services_mentioned": ["payments-svc"],
  "is_evidence_for_question_ids": ["q-1", "q-3"]
}
```

`manifest.json` (run receipt):
```json
{
  "run_id": "...", "profile_name": "...", "seed": 42,
  "started_at": "...", "finished_at": "...", "customer_id": "...",
  "repos": [{"url": "...", "sha": "...", "mode": "hybrid"}],
  "world_model": {"people_count": ..., "services_count": ..., ...},
  "archetypes_executed": {
    "INCIDENT": {"requested": 8, "generated": 8, "dropped": 0, "regen_rounds": 1}
  },
  "totals": {
    "scenarios": ..., "documents": ..., "questions": ...,
    "llm_calls": {"planner": ..., "writer": ..., "validator": ...},
    "cache_hit_rate": ...,
    "cost_estimate_usd": ...
  },
  "warnings_count": ...
}
```

A richer structured `eval_set.json` (per-question scoring config,
retrieval-only ground truth, multi-turn support, prerequisites, lineage)
is **future work** — bolted on when eval-scoring infrastructure needs it.

## 10. Determinism, caching, cost ceiling

### 10.1 Caches

| Cache | Key | Storage | TTL |
|---|---|---|---|
| Repo extraction | `(repo_url, latest_sha)` | `~/.cache/prbe-synth/repos/` | until SHA changes |
| WorldModel | `hash({repo_url: sha}) + hash(company_context)` | `~/.cache/prbe-synth/worldmodel/` | derived |
| LLM responses | `(model, prompt_hash)` | `~/.cache/prbe-synth/llm/` | indefinite; `--no-llm-cache` busts |

### 10.2 RNG seeding

`profile.seed` is the master seed. Component sub-streams:
```
seed_for(component) = sha256(f"{master_seed}:{component}").digest()[:8]
```
Components: `scenarios`, `timestamps`, `persona-sampling`,
`topic-sampling`, `cadence-jitter`, etc.

Same profile + same repo SHAs + warm caches → bit-identical R2 keys, queue
rows, ground truth.

### 10.3 Mock-LLM mode for CI

`--mock-llm` swaps the LLM client for a fixture-backed one. Canned
responses keyed by prompt hash, loaded from `tests/synth/llm-fixtures/`. CI
runs end-to-end without burning Anthropic tokens.

### 10.4 Cost ceiling — soft abort with progress threshold

Two flags:
```
--max-llm-cost-usd 5.00         # budget
--cost-abort-progress 0.30      # only abort if overrun detected
                                # before this fraction of work is done
```

At each LLM call boundary:
```
projected_total = cumulative_cost / max(0.01, work_progress_fraction)
work_progress_fraction = (planner_done + writer_done + validator_done)
                       / projected_total_calls

if projected_total > max_cost AND work_progress_fraction < cost_abort_progress:
    abort with "would have cost ~$X, only at Y% — stopping"
elif projected_total > max_cost:
    warn once, let the run finish
```

`projected_total_calls` is precomputable from the resolved profile
(archetype counts × per-archetype emission counts). Avoids wasting already-
spent tokens on a near-complete run.

## 11. Errors

Three failure classes:

- **LLM call failure** (rate limit / model down / content filter): retry
  with exponential backoff up to 3×. After 3 fails, drop the doc (low-rigor)
  or trigger validator regen (plot). Log to manifest. Run continues.
- **Validation failure on a scenario**: regenerate up to 2× then drop the
  scenario. Run continues with reduced count, logged in manifest.
- **R2 / DB failure**: fail fast. Infrastructure problems; gracefully
  proceeding produces a corrupt run. Crash with partial state cleanly
  identifiable so `--reset` recovers.

### 12.4 Presets shipped with the tool

In `scripts/synth/presets/`:

| Preset | Purpose | Output |
|---|---|---|

## 14. Testing

- **Unit per component**: extractor parsing, WorldModelMerger
  canonicalization, each `SourceWrapper` round-tripping through
  `Connector.parse` with existing `fixtures/<source>/` payloads.
- **Integration small**: a tiny vendored test repo in
  `tests/synth/fixtures/test-repo/` (~10 commits, 3 personas, 2 services).
  `tiny-test` preset against a test customer, asserts queue rows + R2 keys
  + `questions.jsonl` shape.
- **Integration large**: nightly CI run against prbe-knowledge itself in
  `--mock-llm` mode, asserts no regressions in worker drainage rate.

## 15. Open questions / future work

1. **Real-provider replay** (Section 3 mode A from brainstorm) — out of v1
   scope. If we ever want to test real OAuth + webhook signature paths
   end-to-end, we add a new generator at the narrative-layer seam.
2. **Webhook replay to prbe-backend's edge** (mode B) — same seam, future.
3. **Optional WorldModel enrichment from connected providers** (real Slack
   channel names from a connected dev tenant, etc.) — explicitly post-v1
   per scoping discussion. Keep the hook in profile schema but no preset
   uses it.
4. **Richer eval scoring artifact (`eval_set.json`)** — bolt on when eval-
   scoring infrastructure needs it. Would coexist with `questions.jsonl`
   for back-compat with the existing stub runner.
5. **Per-tenant migration to a synth-only DB** — current design puts synth
   tenants in the production DB by `cust-eval-`/`cust-synth-` prefix. If
   pollution becomes a concern, isolate via a separate DB or schema. Not
   needed v1.
6. **Generation parallelism** — v1 runs scenarios sequentially. The
   embarrassingly-parallel parts (writer calls per scenario, recurring
   scenarios) can be parallelized later if wall-clock matters.
7. **Profile composition / inheritance beyond presets** — extending a
   profile from another profile, layered overrides. Not needed v1; presets
   cover the common case.
