## Problem Statement

PRBE is pivoting from an end-to-end debugging product to an **orchestration layer** that makes already-automated debug pipelines trustworthy.

Target customers have already wired Sentry ŌåÆ Linear ŌåÆ coding agent end-to-end. The automation runs. The problem is what comes out of it:

- Agents generate fixes with no knowledge of the company's roadmap, active PRs, related recent work, or conventions.
- Reviewers can't efficiently check whether a generated fix is safe to merge ŌĆö the context needed to say "yes, ship" lives scattered across Slack, Notion, GitHub, Linear.
- Result: automated fixes that nobody trusts. The pipeline produces PRs faster than humans can vet them, so either unvetted fixes merge (regressions, conflicts, architectural violations) or vetted-conservatively fixes get rejected (agent was noise).

The real job-to-be-done is not to automate the chain ŌĆö it's already automated. The job is:

> At the moment an agent picks up a SentryŌåÆLinear ticket, inject relevant company context into the agent's working prompt so the generated fix is informed by roadmap, design docs, active PRs, recent related decisions, and team conventions. Then, at the moment the agent produces a PR, verify the diff against the same context and produce a machine-readable trust assessment: "this fix is merge-safe because X, Y, Z" or "this fix conflicts with PR #1234 / regresses bug-456 / violates convention in runbook RB-012."

That is the orchestration product. PRBE sits at two insertion points in the existing automated chain ŌĆö context-injection at agent-launch, verification at PR-draft ŌĆö and makes the automation actually worth trusting.

**Phase 0** ships the foundational knowledge layer that everything above depends on. Not the orchestration (Phase 1+). Not the verification (Phase 2+). Not the compiled wiki of stable entities (Phase 0.5). **ACL is captured and maintained but not enforced at query time in Phase 0** ŌĆö `documents.acl` + `acl_snapshots` populate from every handler's `extract_acl()`, but the retrieval service's `acl.py` is a pass-through stub. Phase 1 flips enforcement on without a backfill, because the data is already there. Just: ingest customer data from five sources, store normalized + versioned + ACL-captured (unenforced), return ranked chunks via HTTP on query. This is infra with no standalone customer value, built against the thesis that the context-injection and verification products that consume it are what customers will pay for.

## Status Quo

The chain is ALREADY automated end-to-end at target customers. The problem is that the output is untrustworthy.

What actually happens today:

1. Sentry fires, detects a new error fingerprint, auto-creates a Linear ticket via Sentry's native Linear integration. The ticket has the stack trace, the error fingerprint, first_seen/last_seen, affected release ŌĆö Sentry's intra-platform signals only.
2. Linear notifies the relevant team (oncall, service owner). At the same moment, a coding agent (Claude Code / Cursor / custom MCP client / Devin-class autonomous agent) picks up the ticket and starts generating a fix.
3. The agent produces a diff. It's drafted against nothing but the stack trace and what the agent can infer from the codebase. The agent doesn't know:
   - Which active PRs the change might conflict with
   - That the same symptom was fixed two weeks ago in a different subsystem and this fix would regress it
   - That the touched code path is being rewritten as part of the auth migration shipping next sprint
   - That this service is owned by a team whose conventions say "never catch bare exceptions"
   - That the design doc for the affected feature explicitly forbids this approach
4. The agent's fix lands in a PR. Now an engineer has to decide whether to merge it. They face the same context problem in reverse: "is this safe to ship?" To answer, they have to manually check:
   - Is this fix in conflict with any open PR? (greps GitHub)
   - Did we already fix this and it regressed? (searches Slack, commit history)
   - Does this break a planned feature? (opens Notion roadmap, asks around)
5. Most of the time the reviewer skips most of step 4 because it's too much work. So either:
   - The agent's fix gets merged without real review ŌåÆ occasional regression, conflict, or architectural violation ŌåÆ bug fixed, two new bugs created
   - The reviewer rejects the fix conservatively ŌåÆ agent was useless, engineer writes fix manually anyway ŌåÆ automation was noise
   - The reviewer punts review ŌåÆ PR sits for days ŌåÆ the original incident lingers

**The workaround today:** the automated pipeline produces fixes faster than anyone can trust, so the trust gap eats most of the value. Agents operate in the dark, humans rubber-stamp or reject in the dark, and the tooling investment compounds anxiety rather than velocity. That's the real cost being paid ŌĆö not manual labor, but **untrusted automation**.

**What the pitch is NOT:** "a knowledge base for your coding agent." That's what Glean sells, and we should explicitly tell Reevo to buy Glean if that's all they want. Our pitch is: *"Devin already generates PRs you can't trust; we make them shippable by enriching the Linear ticket with company context before Devin runs, and verifying the resulting diff doesn't conflict/regress/break roadmap before anyone merges."* Phase 1 + Phase 2, not Phase 0.

**Phase 0 does NOT ship a wedge for standalone sale.** Phase 0 is pure infrastructure. The wedges live in Phase 1+:

- **Phase 1 wedge (context injection): ticket enrichment worker.** When a SentryŌåÆLinear ticket is created, append a context block to the ticket body before the coding agent picks it up: services involved, owners, related recent PRs, linked design docs, conventions for the touched code paths, recent decisions relevant to the symptom. The agent reads the enriched ticket and produces a fix grounded in company context instead of the stack trace alone. This is the smallest thing a customer pays for, and the one most directly wired to the pain they have now.

- **Phase 2 wedge (verification): MCP server + verification pipeline.** Expose tools to coding agents and CI pipelines: `get_ticket_context(ticket_id)` pulls pre-compiled context; `verify_fix(diff)` runs checks against active PRs (conflict), recent bug fixes (regression), roadmap items (feature break), and conventions (style/architecture); `explain_verdict(verification_id)` produces the human-readable trust assessment. Reviewers see "safe to merge because X, Y, Z" or "do not merge because conflicts with PR #1234" and move in seconds instead of hours.

- **Phase 3 wedge (closed loop): autonomous merge gates and post-merge monitoring.** Verification isn't just advisory ŌĆö it gates merge at the GitHub Actions level. If the agent's fix passes verification, auto-approve and merge behind a feature flag. If it fails, auto-route back to the agent with the specific conflict/regression context for another iteration. Close the loop with post-merge Sentry watch: did the fix actually resolve the error fingerprint?

Phase 0's job is only to make Phase 1 possible, fast.

## Competitive Landscape

### The knowledge-base-alone category is already commoditized

Several mature products ship what Phase 0 would deliver as a standalone: enterprise-wide permissioned search over Slack/GitHub/Linear/Notion/etc., exposed to coding agents via MCP.

| Product | What they ship | Where they win | Where they don't go |
|---|---|---|---|
| **Glean** | 100+ native connectors, permissioned MCP server, works with Claude Code / Cursor / Devin / 20+ MCP hosts. Customers: Databricks, Duolingo, Grammarly, Pinterest, Plaid. | Breadth, maturity, enterprise polish, horizontal coverage. | No ticket-enrichment workflow, no diff verification, no SentryŌåÆagent flow integration, no post-merge monitoring. Sentry requires a custom connector. |
| **Microsoft Copilot (Graph)** | Search + agents across M365 + connectors. | Bundled with M365, default for enterprises already on Microsoft. | Same orchestration gap as Glean. |
| **Atlassian Rovo** | Search + agents across Atlassian products + connectors. | Default for Jira+Confluence shops. | Same orchestration gap. |
| **Hyper (YC batch)** | "Shared brain" for 2-10 person AI-native teams, passive read/write per-message. | Small-team ICP fit. | Pre-traction, horizontal, not vertical debug-focused. |
| **Glean-style entrants** | Will continue to emerge. | N/A | All share the same structural gap: they are retrieval products, not orchestration products. |

### Why PRBE is not a Glean substitute

**PRBE is not selling enterprise search.** If a customer's need is "our coding agent should have access to our knowledge," they should buy Glean. We should actively tell them so. That's not PRBE's product.

PRBE sells orchestration at two specific insertion points:

1. **Ticket enrichment at agent-launch** (Phase 1) ŌĆö injecting context INTO a Linear ticket before the agent reads it. This is a write action into the customer's existing workflow, not a search endpoint a customer would query.
2. **Verification at PR-draft** (Phase 2) ŌĆö reasoning over code diffs against company context to produce a trust verdict. This is not retrieval; this is judgment under a specific constraint shape.

Neither of these is in Glean's product surface. Neither is in Copilot's. Neither is in Rovo's. None of the horizontal search platforms build specialized debug-loop integrations because their ICP is "all of work," not "engineering teams with Devin-class agents."

### Explicit decision: do NOT integrate with Glean as a knowledge backend

We considered building PRBE as a thin orchestration layer on top of Glean's MCP / Client API (skip Phase 0, ship Phase 1 faster). **Rejected.** Reasons:

- **Vendor dependency on a larger competitor during a pivot.** Glean could absorb our orchestration features (add a "conflict detection" agent) on a quarterly timeline. Building on them gives them perfect visibility into what we're building.
- **Pricing tier mismatch.** Glean is enterprise-sold, and their per-seat pricing would stack on top of ours ŌĆö making our offering non-competitive for the mid-market AI-native customers who are our actual ICP.
- **We do not own our data model.** Glean's document model is optimized for search; ours is optimized for orchestration (`FIRES_IN`, `CONFLICTS_WITH`, `REGRESSES` edges, wiki doc_classes). Those shapes can't be retrofitted onto a rented substrate.
- **Valuation / acquisition story.** "Thin orchestration layer on Glean" is a feature. "End-to-end autonomous-debug trust platform with proprietary knowledge graph" is a company.

### What changes if Glean ships a verification layer tomorrow

The verification moat (Phase 2) is the real defensible thing. If Glean ships it first, PRBE's thesis weakens materially. Speed matters. The build-order decision should favor getting Phase 1 + 2 in front of a customer as fast as possible, not polishing Phase 0 infrastructure.

**Monitor signals:**
- Quarterly scan of Glean's product announcements, especially any "code review" or "agent verification" features
- Watch Glean's customer logo list for debug-first AI companies (if they start landing our ICPs, threat escalates)
- Watch Hyper (same YC batch) for vertical-debug moves ŌĆö they're less mature but move faster

## Constraints

- **Timeline:** 6 weeks. Weeks 1-5 are build; Week 6 is a buffer for real-data fixes + onboarding. Anything projecting past Week 6 is a signal to cut scope, not extend.
- **Team:** solo-leveraged with Claude Code + gstack. Optimizing for AI-parallel execution rather than human-serial.
- **Existing stack:** legacy PRBE (`prbe-backend`, `prbe-middleware`, `prbe-mcp-internal`, `prbe-dashboard`, `prbe-electron-sdk`, `prbe-swift-sdk`) stays in maintenance mode. One non-engaged customer kept alive on Fly.io at trivial cost. New Phase 0 work goes into fresh repos, copy-forward from legacy by inspection (read old code, write new code), no imports.
- **Stack locked:** Direct-SDK integrations per source (Slack SDK, Linear GraphQL, PyGithub, notion-client, Sentry REST) ŌĆö NO integration-vendor middleman. Paragon was evaluated and rejected for Phase 0: it saves ~2-3 days of OAuth/token-refresh plumbing but forces a hybrid architecture where half each handler is Paragon-mediated and half is direct-API (Sentry fingerprint filtering, GitHub CODEOWNERS fetching, Notion permission API, Slack channel membership all need direct calls regardless). Unified direct-SDK framework is cleaner and stays consistent with Langfuse/Pylon/Datadog handlers in Phase 0.5+. Reassess at Phase 2+ if customer onboarding UX scales to 20+ tenants. Postgres (Neon Scale tier, Postgres 16, us-east-1) with pgvector extension. Graph modeled as two relational tables (`graph_nodes`, `graph_edges`) with RLS for tenant isolation ŌĆö Apache AGE was evaluated but is not available on Neon. Cloudflare R2 for raw blob storage (bucket-per-tenant), chosen over AWS S3 primarily because R2's free egress eliminates the reprocess-sweep cost (5TB re-read ~$18 on R2 vs ~$472 on S3). FastAPI + asyncio. OpenAI **`text-embedding-3-large` at full 3072 dimensions** stored as pgvector `halfvec(3072)` (HNSW-indexable up to 4000 dims in pgvector 0.7+; Neon ships 0.8). Embedding API cost is 5├Ś `text-embedding-3-small` ($0.13/1M input tokens vs $0.02/1M) ŌĆö charge is by input tokens, independent of output dimensions. Claude Haiku for entity extraction. `botocore<1.36.0` pinned in `pyproject.toml` due to known R2 incompatibility with post-1.36 default checksum behavior.
- **Deploy target:** Fly.io (two apps: ingestion, retrieval). MinIO locally.
- **Budget:** OpenAI + Anthropic API costs per customer must be observable and per-customer attributable from day 1.

## Premises (agreed)

1. Real job-to-be-done is trustable automation: inject company context into agent-bound tickets before the agent runs, and verify the agent's resulting diff before a human merges. The Sentry ŌåÆ Linear ŌåÆ agent chain is already automated at target customers; the missing pieces are context-injection and verification. PRBE sits at those two insertion points.
2. Phase 0 is infra-only by design. Knowledge layer has **zero** standalone customer value ŌĆö any customer who just wants "AI agent reads company knowledge" should buy Glean, not PRBE. The knowledge base is not the product; it is the necessary foundation for Phase 1 (ticket enrichment) and Phase 2 (verification), which ARE the products. Sales pitch is Phase 1+ orchestration; Phase 0 is build-only-never-sold.
3. Five sources required for Phase 0 (Slack, Linear, GitHub, Notion, Sentry). Notion and Slack are not flow-critical but are verification-critical ŌĆö company roadmap lives in Notion, planning conversations in Slack.
4. The long-term moat is verification, not retrieval. Glean/Notion AI/Pinecone commoditize retrieval. "This fix doesn't conflict with PR #1234, doesn't regress bug-456, doesn't break the in-flight auth rewrite" is the defensible product.
5. No named pilot for Phase 0. Infra built on thesis confidence. Pilot attached during Phase 1.
6. Clean break from legacy PRBE. New repos (`prbe-knowledge/`, later `prbe-orchestration/`). Existing `prbe-*` in maintenance mode, one non-engaged customer.
7. Copy-forward, don't import. Legacy code is reference material, not dependency.
8. Stack choices locked. Direct-SDK integrations (no Paragon or other integration vendor), Neon Postgres + pgvector, relational graph model (two tables + RLS) in place of Apache AGE, **OpenAI `text-embedding-3-large` at 3072 dims stored as `halfvec(3072)`**, Haiku, FastAPI. Swap on evidence during build, not speculation.

## Approaches Considered

### Approach A: Vertical-First (spec's recommended path)

Build Slack end-to-end before any other source. `/query` works for Slack at week 2. Replicate the pattern for 4 other sources one at a time. Ship week 8.

- **Pros:** early demo-able pipeline; low week-1 risk
- **Cons:** canonical schema over-indexes on Slack's message-thread shape; rewrites likely around week 4-5 when GitHub PR diffs force revision

### Approach B: Horizontal-First

All 5 handlers complete before `/query` starts. Cleanest final schema but nothing demo-able until week 6. Ship week 8.

- **Pros:** schema designed against all 5 source shapes simultaneously
- **Cons:** zero runtime validation of full pipeline until late; brutal for morale and pilot pitching

### Approach C: Schema-First + Parallel Handler Build Ōśģ CHOSEN

Week 1 lock the canonical schema against all 5 source API specs + fixtures. Week 2 parallel subagent handler build. `/query` working by week 4. Ship week 6.

- **Pros:** leverages AI-parallel execution; earliest `/query`; schema designed holistically; 2 weeks faster than A or B
- **Cons:** requires real week-1 discipline reading 5 API specs; parallel subagent output needs careful integration review; each source's actual webhook payloads may differ subtly from its documentation (usually the "undocumented field appears in real traffic" category)

### Graph schema (relational model, RLS-isolated per customer)

Apache AGE was the initial design choice but is not available on Neon Scale tier (verified via `CREATE EXTENSION age;` failure). The graph is modeled as two standard Postgres tables with Row Level Security for tenant isolation. Operationally this is actually stronger than AGE's per-customer graph pattern ŌĆö RLS is DB-enforced, not app-enforced.

### Error & Rescue Registry (summary)

Named exception classes + retry/DLQ policy, captured here to keep Week 2 implementation consistent across subagents. Full rescue-action detail lives in `shared/exceptions.py` + per-handler docstrings.

| Codepath | Exception class | Policy |
|---|---|---|
| Webhook signature verification | `SignatureInvalidError` | 401, no retry, alert rate > 1% |
| Webhook replay (stale timestamp) | `SignatureExpiredError` | 401, no retry |
| Source API timeout | `APITimeoutError` | Exponential backoff, max 5 retries, then DLQ |
| Source API rate limit (429) | `RateLimitError` | Respect Retry-After header, backoff + retry |
| Source API auth 401/403 | `TokenExpiredError` | DO NOT retry. Mark integration `status='auth_failed'` + alert customer |
| Source API 5xx | `SourceServiceError` | Backoff retry, max 5, then DLQ |
| Unknown payload shape | `NormalizerSchemaError` | DLQ immediately + alert team (vendor changed shape) |
| Missing required field | `NormalizerFieldError` | DLQ + alert |
| Content exceeds max size | `ContentTooLargeError` | Truncate + flag `truncated=true` in metadata, continue |
| OpenAI embed timeout | `EmbedTimeoutError` | Backoff retry, max 3 |
| OpenAI embed rate limit | `EmbedRateLimitError` | Respect headers, worker pauses |
| OpenAI embed 5xx/auth | `EmbedServiceError`/`EmbedAuthError` | Retry 3x (5xx) / DLQ + alert (auth) |
| Haiku timeout | `HaikuTimeoutError` | Retry once, then fallback rule-based entity extraction |
| Haiku malformed JSON | `HaikuParseError` | Retry once with stricter prompt, then fallback |
| Haiku empty output | `HaikuEmptyError` | Fallback to rule-based, log at WARN |
| Haiku rate limit | `HaikuRateLimitError` | Backoff + retry |
| Postgres conn pool exhausted | `PoolExhaustedError` | Worker backoff 1s, retry (pool transient) |
| Postgres FK violation | `FKViolationError` | DLQ + alert (data integrity bug) |
| HNSW index lock | `IndexLockError` | Backoff retry, max 3 |
| Graph node duplicate race | `NodeConflictError` | Upsert pattern ŌĆö not actually an error, handle via ON CONFLICT |
| R2 timeout | `R2TimeoutError` | Backoff retry, max 5 |
| R2 checksum reject | `R2ChecksumError` | Investigate botocore pin, alert (config bug) |
| R2 bucket not found | `R2NotFoundError` | Fail fast ŌĆö bootstrap_customer race (caught by sync-before-OAuth fix) |
| R2 auth | `R2AuthError` | DLQ + alert (credentials rotated incorrectly) |
| Retrieval Haiku router timeout | `RouterTimeoutError` | Skip query expansion, proceed with raw query, log WARN |
| pgvector query timeout | `VectorSearchTimeoutError` | Return BM25+graph results only, log |
| Graph traversal error | `GraphTraversalError` | Fall back to vector+BM25 only |

**Catch-all bans:** no `except Exception:` / `catch Exception` anywhere. CI linter (ruff rule BLE001) enforces.

**Native SDK retries:** OpenAI and Anthropic Python SDKs both ship built-in retries. Configure clients with `max_retries=5` at init ŌĆö no custom retry wrapper. Exponential backoff with jitter happens inside the SDK. Our code only handles the terminal failure (all retries exhausted ŌåÆ DLQ or fallback).

```python
openai_client = OpenAI(max_retries=5)
anthropic_client = Anthropic(max_retries=5)
```

### Embedding batch failure handling

OpenAI embedding batches accept up to 100 inputs per call. If a single input causes the whole batch to fail (invalid UTF-8, content too long after normalizer quirk, etc.), recursive half-split isolates the offender:

```
embed_batch(chunks[100]) fails
    ŌåÆ embed_batch(chunks[0..50]) + embed_batch(chunks[50..100]) in parallel
    ŌåÆ if one half fails: recurse on the failing half
    ŌåÆ single-chunk failure: log to failed_chunks table + skip, continue doc
```

Bounds: max recursion depth = log2(batch_size) Ōēł 7. Worst case for a 100-item batch: 7 extra API calls to isolate 1 bad chunk. Acceptable.

### Performance notes (for implementation)

- **Chunk inserts should batch:** use `execute_many` / `COPY FROM` instead of row-by-row inserts. HNSW + GIN index maintenance per row is slow; batch amortizes.
- **HNSW index tuning:** Phase 0 uses pgvector defaults (`m=16, ef_construction=64`). For large bulk-ingests during initial backfill, consider `SET LOCAL hnsw.ef_construction = 200` for quality. Revisit only if retrieval quality suffers at scale.
- **Haiku router cache hit rate:** aim for ~80%+ on warm cache. If below, background job can pre-warm cache for common query patterns once we have query volume data.

### Query patterns

The retriever module (`services/retrieval/retrievers/graph.py`) exposes a clean Python interface regardless of the underlying SQL. 1-hop traversals are plain joins. Multi-hop (if ever needed, not used in Phase 0) uses recursive CTEs.

```python
# 1-hop, what Phase 0 retriever actually does
async def one_hop_from_entity(
    customer_id: str,
    source_label: str,
    source_canonical_id: str,
    edge_type: str,
    target_label: str,
) -> list[GraphNode]:
    sql = """
        SELECT gn_t.node_id, gn_t.label, gn_t.canonical_id, gn_t.properties
        FROM graph_nodes gn_s
        JOIN graph_edges ge  ON ge.from_node_id = gn_s.node_id
                            AND ge.customer_id = gn_s.customer_id
        JOIN graph_nodes gn_t ON gn_t.node_id = ge.to_node_id
                             AND gn_t.customer_id = gn_s.customer_id
        WHERE gn_s.customer_id = $1
          AND gn_s.label = $2
          AND gn_s.canonical_id = $3
          AND ge.edge_type = $4
          AND gn_t.label = $5
    """
    return await conn.fetch(sql, customer_id, source_label, source_canonical_id, edge_type, target_label)
```

If we ever need multi-hop, we swap in a recursive CTE behind the same Python function signature. Callers don't care.

### Handler interface contract (Week 2 parallel build)

Every handler implements a direct-SDK interface against its source's native API. There is NO integration vendor in the middle ŌĆö each handler owns OAuth, webhook signature verification, normalization, and supplementary API fetching.

**Base class choice: `abc.ABC` with `@abstractmethod`, not `Protocol`.** Rationale: 5 subagents implementing 5 handlers in parallel. `abc.ABC` fails at import time if a method is missing; `Protocol` only fails at usage site. ABC gives Week 2 subagents immediate feedback without waiting for integration review.

**Constants: `shared/constants.py` exports `Enum` subclasses, not bare strings.** `NODE_LABELS`, `EDGE_TYPES`, `SOURCE_SYSTEMS`, `DOC_CLASSES`, `WIKI_DOC_TYPES` are all `enum.StrEnum`. Typos get caught at dev time instead of producing silent graph drift.

```python
# services/ingestion/handlers/base.py
from abc import ABC, abstractmethod

# Each of the methods below is decorated with @abstractmethod so missing
# implementations fail at import time rather than at usage site.
class Handler(ABC):
    source_system: str                   # must match one of shared.constants.SourceSystem
    supported_event_types: list[str]     # e.g. ["message", "message.changed", "message.deleted"]

    def verify_webhook_signature(self, headers: dict, body: bytes, secret: str) -> bool:
        """Verify the HMAC signature of an incoming webhook against the shared secret.
        Source-specific: Slack uses v0 + X-Slack-Signature, GitHub uses X-Hub-Signature-256,
        Linear uses Linear-Signature, etc.
        Returns True if the request is authentic.
        """
        ...

    def parse_webhook_event(self, body: bytes) -> WebhookEvent:
        """Parse the raw HTTP body into a structured event envelope (event_type + payload).
        Source-specific parsing; no normalization yet.
        """
        ...

    async def fetch_supplementary(self, token: str, event: WebhookEvent) -> dict:
        """Make direct source-API calls needed to enrich the event before normalization.
        Examples:
          Slack:   conversations.members for channel ACL, users.info for author
          GitHub:  GET /repos/{r}/contents/CODEOWNERS on push events
          Notion:  GET /pages/{id}/permissions for effective ACL
          Linear:  issue-relation query if webhook didn't include it
          Sentry:  stack-trace top-frame extraction if filtered payload omits it
        Returns a dict of supplementary fields used by normalize() and extract_acl().
        """
        ...

    def normalize(self, event: WebhookEvent, supplementary: dict) -> list[Document]:
        """Convert a source-native event + supplementary data into one or more canonical Documents.
        Pure function. No IO. Deterministic for same inputs.
        """
        ...

    def extract_acl(self, event: WebhookEvent, supplementary: dict) -> list[ACLPrincipal]:
        """Extract ACL principals (users/groups/channels/workspaces) that apply to this content.
        Phase 0 INGESTS and STORES this (in documents.acl + acl_snapshots) but does NOT
        enforce it at query time. Phase 1 enables the filter; data will already be present.
        """
        ...

    async def backfill_paginated(
        self, token: str, since: datetime
    ) -> AsyncIterator[WebhookEvent]:
        """For historical backfill and drift correction: walk the source's list APIs
        with pagination, yielding synthetic WebhookEvents in the same shape as live webhooks.
        normalize() + extract_acl() run on these identically to live events.
        """
        ...
```

A handler passes the week-2 integration gate when:
1. `normalize()` round-trips all committed fixtures in `fixtures/{source}/` without raising
2. `tests/handlers/test_{source}.py` passes with >90% coverage
3. `extract_acl()` returns correct principal set for a hand-annotated fixture subset (enforcement is off, but the data it captures must be correct ŌĆö Phase 1 depends on it)
4. `verify_webhook_signature()` rejects tampered payloads and accepts authentic ones
5. `fetch_supplementary()` handles rate-limit/auth errors gracefully (retry with backoff, escalate on non-transient failures)

### Topology

Sequential within a customer, parallelizable across customers. Different customer = separate Fly task if we ever need to scale horizontally. Within one customer, the Sonnet regeneration + write-back path is strictly sequential with a Postgres advisory lock per wiki page `doc_id` ŌĆö defense in depth against manual-trigger / retry / reprocess overlap.

Haiku triage stays parallel (`asyncio.gather`): it's read-only, no write race possible, and it's ~600ms per candidate ŌĆö parallelizing saves real wall time with zero risk.

### Five-stage pipeline

```
STAGE 1 ŌĆö Collect candidates (for one customer)
  SELECT events FROM ingestion_events
    WHERE customer_id = $1
      AND received_at >= yesterday_midnight
      AND status = 'processed';
  Expand each event to doc_ids it produced/updated.
  For each doc, query graph: which wiki pages are linked to this doc's entities?
  COALESCE candidates by wiki page:
    output = { wiki_page_doc_id: [triggering_event_ids, ...] }
  This is the ONLY dedup step ŌĆö after this, at most one compile per page per batch.

STAGE 2 ŌĆö Haiku triage (parallel, ~$0.0001 per page, ~600ms each)
  asyncio.gather over all candidates:
    prompt_haiku(current_page_content, aggregated_changes_summary)
      ŌåÆ {"needs_update": bool, "reason": str}
  Filter to pages where needs_update=true. Drop the rest silently.

STAGE 3+4 ŌĆö Sonnet regeneration (SEQUENTIAL per customer, ~$0.10-0.20 per page)
  For each surviving page, one at a time:
    SELECT pg_try_advisory_xact_lock(hashtext('wiki:' || doc_id));
    If lock not acquired: skip this page, log, continue (picked up next batch).
    Otherwise:
      - Read ALL current source docs feeding this page (graph traversal)
      - Read current wiki page content (for style continuity, NOT as ground truth)
      - Prompt Sonnet to FULLY REGENERATE the page from sources ŌĆö not incremental edit.
        Rationale: incremental drift accumulates hallucinations over N cycles;
        full regen is drift-free, cost delta is minor at Phase 0.5 scale.
      - Validate: page must cite every claim, format matches wiki.service_card template
      - Compute ACL = intersection of source doc ACLs (if any source is private,
        compiled page inherits the restriction)
      - INSERT into documents as new version (same doc_id, version+1):
          doc_class = 'compiled_wiki'
          doc_type  = 'wiki.service_card'
          supersedes_doc_id = prior version's doc_id (same doc_id, bumped version)
          compiled_from_doc_ids = [list of source doc_ids]
          compilation_model = 'claude-sonnet-4-6'
          compiled_at = NOW()
          compile_trigger = 'scheduled' | 'manual' | 'normalizer_reprocess'
      - Re-chunk + re-embed the new page (same pipeline as ingestion handlers)
      - Upsert graph nodes/edges:
          graph_nodes: upsert ServiceCard node (label='ServiceCard', canonical_id from service)
          graph_edges: upsert DESCRIBES edge (ServiceCard ŌåÆ Service)
          graph_edges: upsert COMPILED_FROM edges (ServiceCard ŌåÆ each raw_source Document)
    Advisory lock releases at transaction commit.

STAGE 5 ŌĆö Observability
  emit per-customer metrics:
    compile_duration_s, pages_updated, pages_skipped_triage, pages_skipped_lock,
    sonnet_tokens_input, sonnet_tokens_output, compilation_cost_usd
```

## Open Questions

1. ~~**S3 vs R2.**~~ **RESOLVED: Cloudflare R2.** Decision-defining reason: R2's free egress turns the normalizer-reprocess-sweep scenario from a ~$470 event (S3) into a ~$18 event (R2). Steady-state storage is ~35-40% cheaper on R2 at every scale point we projected. We have no AWS adjacency (no Lambda/SNS/Athena), so S3's ecosystem premium buys nothing. Migration asymmetry also favors starting on R2: R2ŌåÆS3 at 5TB is ~$105 in egress vs S3ŌåÆR2 being ~$710 ŌĆö if we ever need to move, R2 leaves cheap. Tradeoffs accepted: no native bucket versioning (we content-address at the app layer via `content_hash`, so this was never load-bearing); no cross-region replication (single-region Phase 0 per principles, can add rclone mirror later); narrower event-notification surface (nice-to-have, deferred); slower small-object PUT latency (~340ms p90 vs S3 ~38ms ŌĆö inside our async worker boundary, invisible to the upstream source webhooks since we 200 back within 500ms and persist raw out-of-band).
